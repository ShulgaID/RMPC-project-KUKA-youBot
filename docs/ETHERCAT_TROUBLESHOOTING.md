# Решение проблемы с EtherCAT на KUKA youBot

Этот документ описывает, **почему** драйвер youBot постоянно падает на EtherCAT
и **как** это устранить. Разбор сделан по коду драйвера в этом репозитории
(`src/youbot_driver/src/youbot/EthercatMasterWithoutThread.cpp` и
`EthercatMasterWithThread.cpp`) и конфигу `config/youbot-ethercat.cfg`.

---

## 1. Откуда берётся ошибка

Драйвер инициализирует EtherCAT в методе `initializeEthercat()`. Там есть ровно
три места, где он бросает исключение и роняет ноду:

| Сообщение драйвера | Что произошло |
|---|---|
| `No socket connection on eth0` / `Excecute as root` | SOEM не смог открыть RAW-сокет на сетевом интерфейсе. Либо имя интерфейса неверное, либо нет прав. |
| `No EtherCAT slaves found!` | Сокет открылся, но на линии не нашлось ни одного slave (контроллеров моторов). Физика/питание/кабель. |
| `Not all EtherCAT slaves reached operational state.` | Slave'ы найдены, но не дошли до состояния OPERATIONAL (часто — таймаут цикла, питание базы, distributed clock). |

Конфиг сейчас минимальный и хардкодит интерфейс:

```
EthernetDevice = eth0
CycleTime = 1000
```

В 95 % случаев «постоянная» ошибка — это **первая** строка таблицы, и причина
одна из двух: неправильное имя интерфейса или отсутствие прав на RAW-сокет.

---

## 2. Причина №1 — имя интерфейса не `eth0`

На современных Ubuntu (18.04+) ядро использует *predictable network interface
names*: `enp2s0`, `eno1`, `enx00...` и т.п. Интерфейса `eth0` физически нет,
поэтому `ec_init("eth0")` всегда возвращает 0 → `No socket connection on eth0`.

**Проверить, как реально называется проводной интерфейс к роботу:**

```bash
ip link            # или:  ls /sys/class/net
```

Найдите интерфейс, через который идёт связь с роботом (тот, что в подсети
робота, обычно 10.0.0.x — на нём вы пингуете `ping 10.0.0.1`).

```bash
ip -br addr        # короткий список интерфейс/состояние/адрес
```

**Исправить конфиг** `config/youbot-ethercat.cfg`, подставив реальное имя:

```
EthernetDevice = enp2s0     # <-- ваше реальное имя из ip link
CycleTime = 1000
```

> Если у вас несколько копий запуска (например, отдельный конфиг в установленном
> пакете в `~/catkin_ws/install` или `/opt/ros/...`), поправьте **тот**, который
> реально читается нодой. Проще всего держать один источник и указывать путь
> к нему параметром запуска драйвера.

---

## 3. Причина №2 — нет прав на RAW-сокет

SOEM открывает `AF_PACKET`/RAW-сокет, для чего нужны привилегии. Запускать ROS
из-под `sudo` — плохая идея (ломает окружение и `~/.ros`). Правильное решение —
выдать capability **бинарю**, который реально открывает сокет.

**Разовая выдача прав исполняемому файлу драйвера** (после каждой пересборки
`catkin build`/`catkin_make` повторить, т.к. бинарь пересоздаётся):

```bash
# найдите бинарь ноды драйвера (пример имени, подставьте свой):
ROSNODE=$(catkin_find youbot_driver_ros_interface youbot_driver_ros_interface 2>/dev/null | head -1)
# или вручную, например:
# ROSNODE=~/catkin_ws/devel/lib/youbot_driver_ros_interface/youbot_driver_ros_interface

sudo setcap cap_net_raw,cap_net_admin+eip "$ROSNODE"
getcap "$ROSNODE"     # проверка: должно показать cap_net_admin,cap_net_raw+eip
```

Чтобы не делать это руками после каждой сборки, добавьте post-build шаг или
держите скрипт-обёртку запуска, который сначала проставляет `setcap`, затем
делает `roslaunch`.

> Альтернатива (хуже): `sudo -E roslaunch ...` с сохранением окружения. Работает,
> но загрязняет права на файлы в `~/.ros`. По возможности используйте `setcap`.

---

## 4. Причина №3 — slave'ы не найдены / не выходят в OPERATIONAL

Если имя интерфейса верное и права есть, но видите
`No EtherCAT slaves found!` или `...reached operational state`:

1. **Питание базы.** Контроллеры моторов EtherCAT питаются от свинцового
   аккумулятора базы. Если база не включена/разряжена — slave'ов на линии нет.
   Проверьте, что база реально запитана (см. инструкцию запуска: основной
   свинцовый аккумулятор).
2. **Кабель и линк.** `ip link` должен показывать интерфейс в состоянии `UP`,
   `LOWER_UP` (линк есть). Дёрганый контакт EtherCAT-кабеля = плавающая ошибка
   «то есть slave, то нет».
3. **Только робот на линии.** EtherCAT-интерфейс должен идти ИСКЛЮЧИТЕЛЬНО к
   роботу. Если на этот же интерфейс прилетает обычный IP-трафик/DHCP — мусорные
   кадры мешают SOEM. Лучше выделенный интерфейс только под робота.
4. **Distributed clock / таймаут.** Лог `no distributed clock is available` сам
   по себе не фатален, но если slave'ы не доходят до OP — увеличьте таймауты в
   коде инициализации (`ethercatTimeout`, ожидание состояний) или поднимите
   `CycleTime`. Параметры читаются из cfg (`EtherCATTimeout_[usec]`,
   `MailboxTimeout_[usec]`), их можно добавить в конфиг (см. ниже).
5. **Порядок включения.** Сначала включаем/будим базу, ждём, потом запускаем
   драйвер. Если драйвер стартует раньше, чем slave'ы поднялись, — гонка и
   ошибка OP.

---

## 5. Расширенный конфиг (рекомендуется)

Добавьте таймауты в `config/youbot-ethercat.cfg`, чтобы их можно было крутить
без пересборки (драйвер их читает, если они присутствуют):

```
EthernetDevice = enp2s0
CycleTime = 1000
EtherCATTimeout_[usec] = 500
MailboxTimeout_[usec] = 20000
```

---

## 6. Чек-лист «всё по порядку»

```bash
# 1) интерфейс существует и поднят
ip -br link

# 2) робот пингуется (база включена, кабель ок)
ping -c3 10.0.0.1

# 3) cfg указывает на ПРАВИЛЬНЫЙ интерфейс
grep EthernetDevice config/youbot-ethercat.cfg

# 4) у бинаря драйвера есть cap_net_raw (после КАЖДОЙ пересборки!)
getcap <путь-к-бинарю-драйвера>

# 5) запуск
roslaunch youbot_driver_ros_interface youbot_driver.launch
```

Если все пять пунктов зелёные, EtherCAT поднимается стабильно. «Постоянная»
ошибка почти всегда — это пункт 3 (интерфейс не `eth0`) или пункт 4 (потеряли
`setcap` после пересборки).

---

## 7. Если запускаете в симуляции (Gazebo)

В Gazebo **EtherCAT не используется вообще** — база управляется плагином
`planar_move` через `/cmd_vel`, рука через контроллеры `ros_control`. Если вам
не нужно железо, запускайте sim-стек (`pick_place_sim.launch`) — там драйвер
youBot и EtherCAT не поднимаются, и эта ошибка не возникает по определению.
EtherCAT актуален только для реального робота (`pick_place_real.launch` +
`youbot_driver_ros_interface`).
